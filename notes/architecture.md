Architecture defines a system's structure and constraints to meet requirements. It is tool-agnostic and sets the rules that govern how the system must behave and what boundaries must be respected. The "why" behind each rule matters: a constraint without a reason is just a restriction. Implementation is the specific code and technology used to build according to those rules. The implementation team has full autonomy over tooling choices, as long as they can demonstrate their approach upholds the architectural constraints.
"Refactoring architecture" means changing the fundamental rules and constraints of the system. This is different from refactoring implementation, which changes the code or tools but keeps the architectural rules intact. Implementation can and should inform architecture — but changing the tools is not the same as changing the rules
